如果你曾经用多种语言发布过软件，你一定深有体会。选一个本地化平台，连接到你的代码仓库，然后把剩下的时间都花在同步管理上。内容发出去，翻译回来，中间某个环节总会出问题。

这种开销——内容在你的仓库之间不断来回——是每个团队使用现有本地化工具必须付出的代价。听起来无关紧要，直到周五晚上六点你在调试为什么某个翻译 PR 搞崩了你的站点构建。

## 一个互联网诞生前遗留下来的设计

大多数本地化平台的设计理念都早于现代开发工作流。翻译记忆库、模糊匹配、人工译者在专有编辑器里工作，靠工具从数据库中推荐相似字符串来辅助。

当翻译还是一个手工、离线的过程时，这些想法是合理的。但公司们把翻译记忆库变成了一种锁定机制。你过去的翻译——你花钱积累的机构知识——存在他们的平台里。换到另一家提供商意味着从零开始，或者花钱买一份永远用不顺的导出文件。

结果就是一个建立在人为摩擦之上的行业。你的内容离开你的仓库，进入一个黑箱，然后按别人的时间表回来。

## 没人谈论的 CI 问题

在 [Tuist](https://tuist.dev)，我们为文档站点和营销站点做本地化。我们用过外部本地化平台，经常发现自己在处理这些工具提交的 Pull Request——它们无法通过我们的自动化检查。

问题是结构性的：外部本地化工具无法运行你的 CI 流水线。它们不知道你的 linter、你的构建步骤、你的链接检查器、你的 frontmatter schema。它们把翻译内容推回你的仓库，然后听天由命。

这真的很让人抓狂。我们收到一批翻译，PR 挂掉 CI，团队里就得有人停下手头的事去修格式问题、语法错误或翻译工具引入的无效标记。本不该是问题的事情，却消耗了真正干活的精力。

## 一次改变一切的对话

l10n 的想法来自与 [Maria Jose](https://github.com/mjsesalm) 的思考和交流，她在本地化方面的经验对于让想法落地至关重要。她对语言学和工具链的痛点都有深刻理解。

我们一起问了一个简单的问题：**如果本地化 Agent 能使用开发者使用的同样工具会怎样？**

不是一个来回同步文件的平台。不是一个与代码库脱节的网页编辑器。而是能够访问你的项目、你的环境、你的验证工具的 Coding Agent。这些 Agent 可以翻译文件、运行 linter、检查输出，如果出错就重试——所有这些都在提交 Pull Request 之前完成。

这个想法是让本地化更靠近开发发生的地方，消除中间环节。只有你的项目、一个环境，和一个编排工作的 CLI。

## l10n 是我们送给软件行业的礼物

我们构建 l10n 是因为我们希望更多的软件被本地化，而不是更少。

复杂的流程和昂贵的平台让小团队、独立开发者和开源项目难以企及本地化。如果你的翻译工作流需要走采购流程、按字数谈判价格，还要项目经理来协调交接，大多数团队就会只发布英文版然后收工。

l10n 是开源的。它使用你已经能用的模型。它在你的终端里运行。它用你自己的工具来验证输出，而不是我们的。

我们认为本地化应该像运行测试套件一样自然。

## 下一步：人工输入体验

今天，l10n 是一个为开发者设计的 CLI 工具。但我们知道，并非所有对本地化质量有贡献的人都是开发者。

我们想探索人工输入体验可以是什么样子，就像 OpenAI 用桌面应用为 Codex 把终端抽象化一样。在这种体验中，语言学家不需要考虑分支、编译或 JSON。他们并排看到内容、上下文和翻译。他们优化需要优化的部分。工具处理其他一切：提交、验证、创建 Pull Request。

终端是正确的起点，因为最难的问题要先在这里解决。但终点是一种欢迎所有关心让软件会说每种语言的人的体验。

## 开始使用

用 [mise](https://mise.jdx.dev) 安装 CLI：

```bash
mise use github:tuist/l10n
```

然后初始化你的项目：

```bash
l10n init
```

这会在你的仓库根目录创建一个 `L10N.md` 文件，你可以在其中配置 LLM 提供商、源文件和目标语言。配置完成后，用以下命令翻译内容：

```bash
l10n translate
```

用 `l10n status` 检查哪些内容过期了，用 `l10n check` 验证输出。就这些。不需要注册平台，不需要管理仪表板，不需要和供应商谈判。

l10n 完全免费且开源。如果它感觉像一个打磨过的产品，那是有意为之。在 [Tuist](https://tuist.dev)，我们把开源工作也当作产品对待。我们对 l10n 投入了与我们发布的所有东西同样的用心、同样的开发者体验关注和同样的质量标准。我们相信开源值得这样的工艺水准，依赖它的团队也是如此。
