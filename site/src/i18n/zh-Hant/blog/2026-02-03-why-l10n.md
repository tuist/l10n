如果你曾經發布過多語言軟體，你一定知道這套流程。你選一個本地化平台，連接到你的 repository，然後把剩下的時間都花在管理同步上。內容送出去，翻譯回來，然後在某個環節出問題。

這種開銷——內容在你的 repository 之間不斷來回——是每個團隊使用現今本地化工具所付出的代價。聽起來是小事，直到你在週五下午六點 debug 為什麼某個翻譯 PR 讓你的網站 build 壞掉。

## 一個在網路出現前就定型的設計

大多數本地化平台的設計理念早於現代開發流程。翻譯記憶庫、模糊匹配、人工譯者在專有編輯器裡工作，輔以從資料庫建議相似字串的工具。

這些想法在翻譯還是手動離線作業的年代很合理。但公司把翻譯記憶庫變成了鎖定機制。你過去的翻譯、你付費累積的機構知識，都存在他們的平台裡。換到其他供應商意味著從零開始，或者付費匯出但永遠不太順利。

結果就是一個建立在人為摩擦上的產業。你的內容離開你的 repo，進入一個黑盒子，然後按照別人的時程表回來。

## 沒人談論的 CI 問題

在 [Tuist](https://tuist.dev)，我們本地化我們的文件網站和行銷網站。我們使用過外部本地化平台，經常發現自己在處理這些工具開出來卻無法通過自動化檢查的 pull request。

問題是結構性的：外部本地化工具無法執行你的 CI pipeline。它們不知道你的 linter、你的 build 步驟、你的 link checker、你的 frontmatter schema。它們把翻譯內容推回你的 repo，然後聽天由命。

這真的很令人沮喪。我們收到一批翻譯，PR 在 CI 失敗，團隊裡就得有人放下手邊的工作，去修復翻譯工具產生的格式問題、語法錯誤或無效標記。把精力從真正的工作上抽走，去解決本來就不該是問題的事情。

## 一場改變一切的對話

l10n 的想法來自與 [Maria Jose](https://github.com/mjsesalm) 的思考和交談，她在本地化方面的經驗對於讓想法成形至關重要。她同時深刻理解語言層面和工具層面的痛點。

我們一起問了一個簡單的問題：**如果本地化 agent 可以使用開發者使用的工具呢？**

不是一個來回同步檔案的平台。不是一個與你的 codebase 脫節的網頁編輯器。而是能夠存取你的專案、你的環境和你的驗證工具的 coding agent。這些 agent 可以翻譯一個檔案、執行你的 linter、檢查輸出，如果出問題就重試——全部在開 pull request 之前完成。

這個想法是讓本地化更貼近開發發生的地方，消除中間人。就是你的專案、一個環境，和一個協調工作的 CLI。

## l10n 是我們給軟體產業的禮物

我們打造 l10n，是因為我們希望有更多軟體被本地化，而不是更少。

複雜的流程和昂貴的平台讓小團隊、獨立開發者和開源專案難以觸及本地化。如果你的翻譯工作流程需要採購程序、按字計價的談判，還需要專案經理協調交接，大多數團隊會直接用英文發布，就這樣算了。

l10n 是開源的。它使用你已經能存取的模型。它在你的終端機執行。而且它用你自己的工具驗證輸出，不是我們的。

我們認為本地化應該像執行測試套件一樣自然。

## 下一步：人工輸入體驗

今天，l10n 是一個為開發者設計的 CLI 工具。但我們知道，不是每個對本地化品質有貢獻的人都是開發者。

我們想要探索人工輸入體驗可以是什麼樣子，就像 OpenAI 用 Codex 的桌面應用程式抽象化了終端機。一種體驗，讓語言專家不需要思考 branch、編譯或 JSON。他們看到內容、上下文和翻譯並排顯示。他們精煉需要精煉的部分。工具處理其他一切：commit、驗證、開 pull request。

終端機是正確的起點，因為最難的問題都在這裡先被解決。但終點線是一個歡迎所有關心讓軟體說每種語言的人的體驗。

## 開始使用

使用 [mise](https://mise.jdx.dev) 安裝 CLI：

```bash
mise use github:tuist/l10n
```

然後初始化你的專案：

```bash
l10n init
```

這會在你的 repository 根目錄建立一個 `L10N.md` 檔案，你可以在其中配置你的 LLM 供應商、原始檔案和目標語言。配置完成後，翻譯你的內容：

```bash
l10n translate
```

用 `l10n status` 檢查哪些內容過時了，用 `l10n check` 驗證輸出。就這樣。不需要註冊平台，不需要管理儀表板，不需要和供應商談判。

l10n 完全免費且開源。如果它感覺像一個精緻的產品，那是刻意的。在 [Tuist](https://tuist.dev)，我們把開源工作也當作產品來對待。我們對 l10n 投入同樣的用心、同樣對開發者體驗的關注，和同樣的品質標準，就像我們發布的所有東西一樣。我們相信開源值得這種程度的工藝，依賴它的團隊也值得。
