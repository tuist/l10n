複数の言語でソフトウェアをリリースしたことがあるなら、この流れはおなじみでしょう。ローカライゼーションプラットフォームを選び、リポジトリに接続し、その後は同期の管理に時間を費やす。コンテンツを送り出し、翻訳が戻ってきて、その間のどこかで何かが壊れる。

このオーバーヘッド、リポジトリとの絶え間ないコンテンツの往復が、今日のローカライゼーションツールを使うすべてのチームが支払う税金です。些細なことに聞こえるかもしれませんが、金曜日の午後6時に翻訳PRがサイトビルドを壊した理由をデバッグしているのが自分だったら、話は別です。

## インターネット以前から受け継がれた設計

ほとんどのローカライゼーションプラットフォームは、現代の開発ワークフロー以前の概念を中心に設計されました。翻訳メモリ。ファジーマッチング。プロプライエタリなエディタ内で作業する人間の翻訳者を、データベースから類似の文字列を提案するツールがサポートする。

これらのアイデアは、翻訳が手動のオフラインプロセスだった時代には理にかなっていました。しかし、企業は翻訳メモリをロックインの仕組みに変えてしまいました。過去の翻訳、つまりお金を払って得た組織の知識は、彼らのプラットフォーム内に存在します。別のプロバイダーに移行するということは、ゼロからやり直すか、決してうまく機能しないエクスポートにお金を払うことを意味します。

その結果、人為的な摩擦の上に成り立つ業界ができあがりました。コンテンツがリポジトリを離れ、ブラックボックスに入り、誰かのスケジュールで戻ってくる。

## 誰も語らないCIの問題

[Tuist](https://tuist.dev)では、ドキュメントサイトとマーケティングサイトをローカライズしています。外部のローカライゼーションプラットフォームを使用していましたが、それらのツールが作成したプルリクエストが自動チェックに失敗する状況に頻繁に直面しました。

問題は構造的なものです：外部のローカライゼーションツールはCIパイプラインを実行できません。リンター、ビルドステップ、リンクチェッカー、フロントマタースキーマのことを知りません。翻訳されたコンテンツをリポジトリにプッシュして、うまくいくことを祈るだけです。

これは本当にフラストレーションでした。翻訳のバッチを受け取り、PRがCIに失敗し、チームの誰かが手を止めて、翻訳ツールが導入したフォーマットの問題、壊れた構文、無効なマークアップを修正しなければなりませんでした。そもそも問題であるべきでないことを解決するために、本来の仕事からエネルギーを奪われていたのです。

## すべてを変えた会話

l10nのアイデアは、[Maria Jose](https://github.com/mjsesalm)との思考と会話から生まれました。彼女のローカライゼーションにおける経験が、アイデアを具体化する上で重要でした。彼女は言語的な側面とツールのフラストレーションの両方を深く理解していました。

一緒に、シンプルな質問をしました：**ローカライゼーションエージェントが開発者と同じツールを使えたらどうなるだろう？**

ファイルを行き来させて同期するプラットフォームではない。コードベースから切り離されたWebエディタでもない。代わりに、プロジェクト、環境、検証ツールにアクセスできるコーディングエージェント。ファイルを翻訳し、リンターを実行し、出力をチェックし、何かが壊れたらリトライできるエージェント。すべてプルリクエストを開く前に。

アイデアは、ローカライゼーションを開発が行われる場所に近づけ、仲介者を排除することでした。プロジェクト、環境、そして作業をオーケストレーションするCLIだけ。

## l10nはソフトウェア業界への贈り物

より多くのソフトウェアがローカライズされることを願って、l10nを作りました。

複雑なプロセスと高価なプラットフォームが、小規模チーム、個人開発者、オープンソースプロジェクトにとってローカライゼーションをアクセスしにくいものにしています。翻訳ワークフローに調達プロセス、単語単位の価格交渉、ハンドオフを調整するプロジェクトマネージャーが必要なら、ほとんどのチームは英語でリリースして終わりにするでしょう。

l10nはオープンソースです。すでにアクセスできるモデルを使用します。ターミナルで動作します。そして、私たちのツールではなく、あなた自身のツールで出力を検証します。

ローカライゼーションは、テストスイートを実行するのと同じくらい自然であるべきだと考えています。

## 次のステップ：人間のインプット体験

今日、l10nは開発者向けに設計されたCLIツールです。しかし、ローカライゼーションの品質に貢献するすべての人が開発者ではないことは分かっています。

OpenAIがCodexのデスクトップアプリでターミナルを抽象化したように、人間のインプット体験がどのようなものになり得るかを探求したいと考えています。言語学者がブランチ、コンパイル、JSONについて考える必要がない体験。コンテンツ、コンテキスト、翻訳を並べて見る。洗練が必要なものを洗練する。そしてツールがそれ以外のすべて、コミット、検証、プルリクエストの作成を処理する。

ターミナルは最も難しい問題が最初に解決される場所なので、正しい出発点です。しかしゴールは、ソフトウェアをあらゆる言語で話せるようにすることに関心のあるすべての人を歓迎する体験です。

## 始めましょう

[mise](https://mise.jdx.dev)でCLIをインストール：

```bash
mise use github:tuist/l10n
```

次にプロジェクトを初期化：

```bash
l10n init
```

これにより、リポジトリのルートに`L10N.md`ファイルが作成され、LLMプロバイダー、ソースファイル、ターゲット言語を設定します。設定が完了したら、以下でコンテンツを翻訳：

```bash
l10n translate
```

`l10n status`で古くなったものを確認し、`l10n check`で出力を検証します。それだけです。サインアップするプラットフォームも、管理するダッシュボードも、交渉するベンダーもありません。

l10nは完全に無料でオープンソースです。洗練された製品のように感じられるなら、それは意図的です。[Tuist](https://tuist.dev)では、オープンソースの仕事も製品として扱っています。l10nにも、私たちがリリースするすべてのものに注ぐのと同じ注意、同じ開発者体験への配慮、同じ品質基準を注いでいます。オープンソースはそのレベルのクラフトに値すると信じていますし、それに依存するチームも同様です。
